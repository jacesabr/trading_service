TradingView Ideas → Demo Execution Engine

Build plan · trading_service · drafted 2026-06-16 · paper / demo only

Turn the world's stream of community trading ideas into a live, broker-executed, self-scoring track record. Every published idea becomes a signal; a real demo account is the honest judge (real fills, real spread, real P&L); the post's engagement (boosts, comments, author) becomes a feature set we test — does the crowd's attention actually predict edge? This is the lab's existing philosophy — collect a signal → execute on a real service → let the DB decide — pointed at a brand-new, effectively unlimited signal source, in realtime: a monitor fires the moment an idea is published, so we trade it on the demo account within seconds.

The chart image is where the trade lives. Listing cards only carry the thesis snippet; the entry / target / stop are almost always drawn on the annotated chart (horizontal lines + price labels, projected path, FVG/fib zones). So step 3–4 must open each idea page and run a vision model on the image — text-only scraping would miss the actual levels. This is the single hardest, must-validate-first part of the build.

3 · Components to build

a · ideas_ingest.py — live monitor + poll

Push, not just pull (researched stack). Firecrawl monitor does job A entirely in code: POST /v2/monitor with schedule (cron or "every 5 minutes"), target = each feed (newest / hottest / editors-picks), goal:"a new idea appears" (an AI judge filters format-only noise), and a webhook.url → fires monitor.page the moment a new post lands, so we process it in seconds (short-TF ideas have a narrow entry window a slow poll would miss). Underneath, a poll backfill with Tavily (POST /extract, include_images:true — tested ✓, returns symbol + chart-image URLs) as the drop-in 2nd, Apify 3rd. Browse AI is the strongest alternative monitor (its taskCapturedDataChanged webhook gives explicit previous-vs-new list diffs) but needs a robot recorded in its UI first; Octoparse (task-based, no API webhooks) is not used here. Dedupe by url.

b · ideas_vision.py (MULTIMODAL extract / generate)

The keystone — and it must read the chart image. Per idea: fetch the full page (text) + the chart image, then one strict-JSON call to a vision-capable model. It reads the drawn levels off the chart (entry/target/stop lines + price labels, projected path → direction, zones), cross-checks against the text, and falls back to generating levels from the thesis + live price + ATR when nothing is drawn. Output tags basis: chart | text | generated so a real author level is never confused with our own. Model: a NIM vision-language model (or Claude/other VLM) — needs validating on real charts first.

c · ideas_router.py

Map a TradingView symbol (EXCHANGE:TICKER) to a venue + instrument, with an honest coverage table. Unsupported markets (e.g. NSE/Indian equities) are recorded but not executed — we still track whether the idea would have worked (sim), just labelled "no demo venue".

d · ideas_exec.py + ideas table

Reuse the bracket-OCO pattern (broker holds the exit). New table stores the social metadata: url, author, symbol, thesis, boosts, comments, published_ts, params, basis, exec_id, outcome, ret_bps. Hooks into the existing executions table for the fills.

4 · The research angle — is the crowd an edge?

The reason to store boosts and comments is a real, testable hypothesis: does engagement predict accuracy? Once we have enough resolved idea-trades we can ask —

  • Author edge: which authors beat random on real fills? (a leaderboard, min-N gated)
  • Boost threshold: do ideas above X boosts win more than below? Or is it contrarian (hyped = crowded = worse)?
  • Explicit vs generated: do author-printed levels beat AI-generated-from-thesis?
  • Symbol / timeframe: where do community ideas actually carry — crypto vs equities, intraday vs swing?

Every one of these is the same leak-safe, let-the-DB-decide discipline the lab already runs — just with social features as the explanatory variables. Disciplined rejection (most ideas are noise) is the expected, honest outcome — the filter is the product.

6 · Safety floors & bounds

  • Demo / paper only. Real money still requires LIVE_BUDGET_ARMED=1 — left off.
  • ≤ 50 concurrent open trades — global hard cap. Doubles as the compute governor: check the open count first; if it's at 50, skip the expensive scrape + vision entirely (no point extracting an idea we can't place). Broker OCO frees slots as trades resolve. No babysitting of running trades — the venue holds the exit.
  • ≤ 4H timeframe only. The vision read includes the chart's TF; ideas above 4H (1D / 1W swing) are dropped — we want intraday/short-swing, not multi-week positions that tie up slots.
  • Per-source/symbol caps: running set of ~10 per source, ≤2 open per symbol; small fixed qty; one independent order per idea (deduped); broker-managed OCO exits so nothing accumulates.
  • AI-generated levels are flagged — never presented as the author's call.

8 · Bonus: thesis → AI-generated idea

Many posts are pure market thesis (no levels) — e.g. "Bitcoin holds firm near $63K as traders await catalysts… the $63–64K range suggests short-term balance." Instead of discarding these, the LLM generates a concrete, executable trade from the thesis + the live chart context (price, recent ATR/zones): direction, entry, a target and stop sized to volatility. We then trade our interpretation of the thesis on the demo account and score it — separately tagged from explicit-level ideas. This turns the entire qualitative-commentary half of the feed into testable signals, and doubles as a measure of how good our own thesis→trade generation is.

9 · Open decisions (your call)

  • One trade per idea — NO consensus. Ideas are on different timeframes/setups, so there is nothing comparable to "vote" on. Each idea is an independent trade: its own direction, entry, target, stop, and timeframe → its own hold. We execute and score each on its own; the broker OCO closes it whenever its TP/SL hits, be that hours or days. A per-symbol cap (e.g. ≤2 open at once) is only a volume guard, not a vote.
  • Timeframe → max-hold: only ≤4H ideas are taken; the chart TF sets a max-hold cap (a 4H idea holds a bounded number of bars) after which we flatten if neither TP nor SL hit. The OCO does the rest.
  • Global cap = 50 concurrent open trades, checked before any expensive work (compute governor). At cap → skip; slots free as OCO exits resolve.
  • Non-US coverage: record-only sim for unsupported markets, or skip entirely? (recommend record-only — it still answers "is the author good?").
  • Generated-level aggression: how wide a stop/target the LLM may set (ATR multiples) — a guardrail so it can't invent absurd brackets.
  • Vision model: NIM VLM vs Claude vs another — pick on a head-to-head accuracy test reading real annotated charts (this gates the whole project).
  • Scraper split: Firecrawl primary / Apify fallback ratio + the 6h cadence (confirmed).
Recommendation — de-risk the chart read first. P1: scrape a real day of ideas, opening each page and saving the chart images + text. P2: run the vision model on ~30 of them and check by hand whether it pulls the right entry/target/stop off the chart. That single test decides the project — if the vision read is reliable, P3 execution is fast (the bracket-order layer already exists); if it's shaky, we adjust (crop to the price axis, prefer text levels, lower confidence → record-only) before spending on execution. Near-zero risk, and it confronts the hardest part immediately.

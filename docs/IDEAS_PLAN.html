TradingView Ideas ‚Üí Demo Execution Engine

Build plan ¬∑ trading_service ¬∑ drafted 2026-06-16 ¬∑ paper / demo only

Turn the world's stream of community trading ideas into a live, broker-executed, self-scoring track record. Every published idea becomes a signal; a real demo account is the honest judge (real fills, real spread, real P&L); the post's engagement (boosts, comments, author) becomes a feature set we test ‚Äî does the crowd's attention actually predict edge? This is the lab's existing philosophy ‚Äî collect a signal ‚Üí execute on a real service ‚Üí let the DB decide ‚Äî pointed at a brand-new, effectively unlimited signal source, in realtime: a monitor fires the moment an idea is published, so we trade it on the demo account within seconds.

The chart image is where the trade lives. Listing cards only carry the thesis snippet; the entry / target / stop are almost always drawn on the annotated chart (horizontal lines + price labels, projected path, FVG/fib zones). So step 3‚Äì4 must open each idea page and run a vision model on the image ‚Äî text-only scraping would miss the actual levels. This is the single hardest, must-validate-first part of the build.

3 ¬∑ Components to build

a ¬∑ ideas_ingest.py ‚Äî live monitor + poll

Push, not just pull. Use Firecrawl's monitor on several feeds at once ‚Äî newest, hottest, editors-picks ‚Äî so a new post fires a webhook the moment it appears (an AI judge filters noise/format-only changes). A small endpoint receives it and kicks off processing immediately ‚Äî important because short-timeframe ideas have a narrow entry window a 6h poll would miss. A periodic poll (Firecrawl primary / Apify fallback, walking /ideas/page-N/?sort=recent_extended) runs underneath as a backfill/safety net so nothing is dropped if a webhook is missed. Incremental: dedupe by url.

b ¬∑ ideas_vision.py (MULTIMODAL extract / generate)

The keystone ‚Äî and it must read the chart image. Per idea: fetch the full page (text) + the chart image, then one strict-JSON call to a vision-capable model. It reads the drawn levels off the chart (entry/target/stop lines + price labels, projected path ‚Üí direction, zones), cross-checks against the text, and falls back to generating levels from the thesis + live price + ATR when nothing is drawn. Output tags basis: chart | text | generated so a real author level is never confused with our own. Model: a NIM vision-language model (or Claude/other VLM) ‚Äî needs validating on real charts first.

c ¬∑ ideas_router.py

Map a TradingView symbol (EXCHANGE:TICKER) to a venue + instrument, with an honest coverage table. Unsupported markets (e.g. NSE/Indian equities) are recorded but not executed ‚Äî we still track whether the idea would have worked (sim), just labelled "no demo venue".

d ¬∑ ideas_exec.py + ideas table

Reuse the bracket-OCO pattern (broker holds the exit). New table stores the social metadata: url, author, symbol, thesis, boosts, comments, published_ts, params, basis, exec_id, outcome, ret_bps. Hooks into the existing executions table for the fills.

4 ¬∑ The research angle ‚Äî is the crowd an edge?

The reason to store boosts and comments is a real, testable hypothesis: does engagement predict accuracy? Once we have enough resolved idea-trades we can ask ‚Äî

  • Author edge: which authors beat random on real fills? (a leaderboard, min-N gated)
  • Boost threshold: do ideas above X boosts win more than below? Or is it contrarian (hyped = crowded = worse)?
  • Explicit vs generated: do author-printed levels beat AI-generated-from-thesis?
  • Symbol / timeframe: where do community ideas actually carry ‚Äî crypto vs equities, intraday vs swing?

Every one of these is the same leak-safe, let-the-DB-decide discipline the lab already runs ‚Äî just with social features as the explanatory variables. Disciplined rejection (most ideas are noise) is the expected, honest outcome ‚Äî the filter is the product.

6 ¬∑ Safety floors (unchanged)

  • Demo / paper only. Real money still requires LIVE_BUDGET_ARMED=1 ‚Äî left off.
  • Bounded: top 500 most-recent idea-trades; small fixed qty; one order per idea (deduped); broker-managed OCO exits so nothing accumulates.
  • AI-generated levels are flagged ‚Äî never presented as the author's call.
  • Per-run cap on new executions so a busy feed can't fan out unbounded.

8 ¬∑ Bonus: thesis ‚Üí AI-generated idea

Many posts are pure market thesis (no levels) ‚Äî e.g. "Bitcoin holds firm near $63K as traders await catalysts‚Ä¶ the $63‚Äì64K range suggests short-term balance." Instead of discarding these, the LLM generates a concrete, executable trade from the thesis + the live chart context (price, recent ATR/zones): direction, entry, a target and stop sized to volatility. We then trade our interpretation of the thesis on the demo account and score it ‚Äî separately tagged from explicit-level ideas. This turns the entire qualitative-commentary half of the feed into testable signals, and doubles as a measure of how good our own thesis‚Üítrade generation is.

9 ¬∑ Open decisions (your call)

  • Timeframe ‚Üí hold/exit: map the idea's chart TF (15m, 4h, 1D) to a max-hold + how we treat "swing" ideas that run days. Default: TF-scaled max bars, broker OCO does the rest.
  • One order vs scaling: single bracket per idea (simple) ‚Äî recommended to start.
  • Non-US coverage: record-only sim for unsupported markets, or skip entirely? (recommend record-only ‚Äî it still answers "is the author good?").
  • Generated-level aggression: how wide a stop/target the LLM may set (ATR multiples) ‚Äî a guardrail so it can't invent absurd brackets.
  • Vision model: NIM VLM vs Claude vs another ‚Äî pick on a head-to-head accuracy test reading real annotated charts (this gates the whole project).
  • Scraper split: Firecrawl primary / Apify fallback ratio + the 6h cadence (confirmed).
Recommendation ‚Äî de-risk the chart read first. P1: scrape a real day of ideas, opening each page and saving the chart images + text. P2: run the vision model on ~30 of them and check by hand whether it pulls the right entry/target/stop off the chart. That single test decides the project ‚Äî if the vision read is reliable, P3 execution is fast (the bracket-order layer already exists); if it's shaky, we adjust (crop to the price axis, prefer text levels, lower confidence ‚Üí record-only) before spending on execution. Near-zero risk, and it confronts the hardest part immediately.
